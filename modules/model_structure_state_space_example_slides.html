Outline

In this module, we'll learn how to query a model about its graph.

model-generic programming = querying a model's graph + operating a model

Specifically, we'll learn:

  • about topological ordering of nodes and why it matters.
  • how to determine the dependencies (or children) of one or more nodes, and
    • what different kinds of dependencies mean.
  • how to expand node names.
  • how to determine node categories such as top-level nodes, latent-state nodes, and data nodes.
  • how NIMBLE sometimes inserts lifted nodes into a model and what that means for you.
  • how to determine the variables in a model.

We first need to build the model. For this module we will use the smaller version of the state-space model, with only the first four time points. The only purpose of making it smaller is to simplify the examples and the graph figures for clarity.

library(igraph)
## 
## Attaching package: 'igraph'
## The following objects are masked from 'package:stats':
## 
##     decompose, spectrum
## The following object is masked from 'package:base':
## 
##     union
ssmSmallCode <- nimbleCode({
# Priors and constraints
logN.est[1] ~ dnorm(5.6, 0.01)       # Prior for initial population size
mean.r ~ dnorm(1, 0.001)             # Prior for mean growth rate
sigma.proc ~ dunif(0, 1)             # Prior for sd of state process
sigma.obs ~ dunif(0, 1)              # Prior for sd of observation process

# Likelihood
# State process
for (t in 1:(T-1)){
   r[t] ~ dnorm(mean.r, sd = sigma.proc)
   logN.est[t+1] <- logN.est[t] + r[t]
   }
# Observation process
for (t in 1:T) y[t] ~ dnorm(logN.est[t], sd = sigma.obs)

})


pyears <- 6 # Number of future years with predictions
hm <- c(271, 261, 309, 318, 231, 216, 208, 226, 195, 226, 233, 209, 226, 192, 191, 225,
        245, 205, 191, 174, rep(NA, pyears))
year <- 1990:(2009 + pyears)

# Bundle data
bugs.data <- list(y = log(hm), T = length(year))
## NIMBLE will handle y as data, T as a constant

ssmSmall <- nimbleModel(ssmSmallCode, constants = list(T = 5),
                        data = list(y = bugs.data$y[1:5]))

Topological ordering of nodes and why it matters.

Suppose we want to change the value of r[3] and update all relevant parts of the graph. (The meaning of "relevant" depends on our purpose of changing r[3]. In this case imagine we are making a Metropolis-Hastings (MCMC) proposal and need to calculate its acceptance probability.) The nodes that need to be updated are shown in red.

## function drawGraph is defined in the R file for this module
drawGraph(ssmSmall, ssmSmall$getDependencies('r[3]'))

Note that order matters. If we calculated the log probability density of y[4] before calculating logN.est[4], then y[4] would use the old value of logN.est[4]. That would be a mistake.

A valid order for calculations is called topologically sorted. Most of NIMBLE's model-querying functions will return nodes in tologically sorted order.

Inspecting the model: Querying graph dependencies

There are several kinds of dependencies that may be needed by different algorithms.

  • Stochastic dependencies (called Markov blanket in computer science) terminate at stochastic nodes (inclusive). That's the default.
ssmSmall$getDependencies('r[3]')
## [1] "r[3]"        "logN.est[4]" "logN.est[5]" "y[4]"        "y[5]"
drawGraph(ssmSmall, ssmSmall$getDependencies('r[3]'))

  • Stochastic dependencies are needed for a Metropolis-Hastings MCMC sampler.

  • Deterministic dependencies terminate at stochastic nodes but omit the stochastic nodes. These can be useful for implementing a particle filter.

ssmSmall$getDependencies('r[3]', determOnly = TRUE)
## [1] "logN.est[4]" "logN.est[5]"
drawGraph(ssmSmall, ssmSmall$getDependencies('r[3]', determOnly = TRUE))
  • Other ways to control the dependency query can be found by
args(ssmSmall$getDependencies)
## function (nodes, omit = character(), self = TRUE, determOnly = FALSE, 
##     stochOnly = FALSE, includeData = TRUE, dataOnly = FALSE, 
##     includeRHSonly = FALSE, downstream = FALSE, returnType = "names", 
##     returnScalarComponents = FALSE) 
## NULL
## or help(modelBaseClass)

Finding out about nodes

The functions model$getNodeNames, model$expandNodeNames, and model$topologicallySortNodes will query the model about nodes and their roles in the model.

Examples include:

  • All nodes in topologically-sorted order.
    • Note that topologically-sorted orderings are typically not unique. NIMBLE determines one and sticks with it.
ssmSmall$getNodeNames()
##  [1] "logN.est[1]" "mean.r"      "sigma.proc"  "sigma.obs"   "r[1]"       
##  [6] "r[2]"        "r[3]"        "r[4]"        "y[1]"        "logN.est[2]"
## [11] "logN.est[3]" "y[2]"        "logN.est[4]" "y[3]"        "logN.est[5]"
## [16] "y[4]"        "y[5]"
  • Stochastic, deterministic, and data nodes (which are also stochastic).
ssmSmall$getNodeNames(stochOnly = TRUE)   #salmon
##  [1] "logN.est[1]" "mean.r"      "sigma.proc"  "sigma.obs"   "r[1]"       
##  [6] "r[2]"        "r[3]"        "r[4]"        "y[1]"        "y[2]"       
## [11] "y[3]"        "y[4]"        "y[5]"
ssmSmall$getNodeNames(determOnly = TRUE)  #cyan
## [1] "logN.est[2]" "logN.est[3]" "logN.est[4]" "logN.est[5]"
ssmSmall$getNodeNames(dataOnly = TRUE)    #plum
## [1] "y[1]" "y[2]" "y[3]" "y[4]" "y[5]"
drawGraph(ssmSmall, 'stochDetermData')
  • Top nodes, latent nodes, and end nodes
ssmSmall$getNodeNames(topOnly = TRUE)   #salmon
## [1] "logN.est[1]" "mean.r"      "sigma.proc"  "sigma.obs"
ssmSmall$getNodeNames(latentOnly = TRUE)  #cyan
## [1] "r[1]"        "r[2]"        "r[3]"        "r[4]"        "logN.est[2]"
## [6] "logN.est[3]" "logN.est[4]" "logN.est[5]"
ssmSmall$getNodeNames(endOnly = TRUE)    #plum
## [1] "y[1]" "y[2]" "y[3]" "y[4]" "y[5]"
drawGraph(ssmSmall, 'topLatentEnd')

Another kind of node is called RHSonly if it only appears on the right-hand side of a BUGS declaration and never on the left-hand side.

You can get the nodes present in some arbitrary range of a variable like this:

ssmSmall$expandNodeNames('logN.est[2:5]')
## [1] "logN.est[2]" "logN.est[3]" "logN.est[4]" "logN.est[5]"

Finally, if for some reason you did not obtain a vector of node names from the model in the first place and need it topologically sorted, you can do that:

ssmSmall$topologicallySortNodes(c('logN.est[5]', 'r[4]', 'logN.est[3]', 'y[2]'))
## [1] "r[4]"        "logN.est[3]" "y[2]"        "logN.est[5]"

The importance of querying a model: lifted nodes.

You should never assume you know what nodes are in a model simply because you wrote the BUGS code.

NIMBLE implements some features by inserting its own nodes.

Let's look at the two main ways this happens:

m1 <- nimbleModel(
    nimbleCode({
        tau ~ dunif(0, 100)
        x ~ dnorm(0, tau) #by default, tau is a precision
    }))
## defining model...
## building model...
## checking model...   (use nimbleModel(..., check = FALSE) to skip model check)
## NAs were detected in model variables: tau, logProb_tau, lifted_d1_over_sqrt_oPtau_cP, x, logProb_x.
## model building finished
plot(m1$getGraph())
m1$getNodeNames()
## [1] "tau"                          "lifted_d1_over_sqrt_oPtau_cP"
## [3] "x"

The node lifted_d1_over_sqrt_oPtau_cP has been inserted between tau and x. The resulting model would equivalently have been created by this BUGS code:

nimbleCode({
    tau ~ dunif(0, 100)
    lifted_d1_over_sqrt_oPtau_cP <- 1/sqrt(tau)
    x ~ dnorm(0, sd = lifted_d1_over_sqrt_oPtau_cP)
}))

NIMBLE has lifted the calculation of standard deviation from precision so that it is part of the model's graph. Therefore you will make a mistake if you assume that the dependencies of tau include only x:

m1$calculate(c('tau','x')) ## Wrong: the lifted node is being neglected

The correct way to do this is with model-generic programming:

m1$calculate( m1$getDependencies('tau') )

The second common situation that creates lifted nodes is this:

m2 <- nimbleModel(
    nimbleCode({
        a ~ dnorm(0, 1)
        b ~ dnorm(a + 1, 1)
    }))
## defining model...
## building model...
## checking model...   (use nimbleModel(..., check = FALSE) to skip model check)
## NAs were detected in model variables: a, logProb_a, lifted_a_plus_1, b, logProb_b.
## model building finished
plot(m2$getGraph())
m2$getNodeNames()
## [1] "a"               "lifted_a_plus_1" "b"

In this case, NIMBLE lifted the expression a+1 and created a node for it. Again the correct way to program is to query a model to learn about its nodes. Do not assume you know what the nodes are.

Querying the sizes and types of model variables

Sometimes it is useful to ask a model about its variables and their sizes. As of the current version of NIMBLE, information about variables is not competely nicely arranged for a user (there aren't as many query functions), but it is available:

ssmSmall$getVarNames()
## [1] "logN.est"   "mean.r"     "sigma.proc" "sigma.obs"  "r"         
## [6] "y"
ssmSmall$getVarInfo('r')
## Reference class object of class "varInfoClass"
## Field "varName":
## [1] "r"
## Field "mins":
## [1] 1
## Field "maxs":
## [1] 4
## Field "nDim":
## [1] 1
## Field "anyStoch":
## [1] TRUE

This variable has 1 dimension (nDim), and its size is 4 (maxs). Currently mins is always 1. If at least one node within the variable is stochastic, then anyStoch will be TRUE.
